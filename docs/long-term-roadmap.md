# 长期规划：个人投资决策与复盘系统

本文档记录 Personal Invest 的长期方向。它不替代 `docs/task-board.md`：

- `task-board.md` 管当前已确认要执行的任务。
- 本文档管业务定位、模型演进方向和未来阶段规划。

## 当前状态

系统已经从“个人投资看板”升级为“个人投资决策与复盘系统”的早期形态。

当前核心闭环是：

```text
观察资产
  ↓
行情 / 净值 / 市场 / 行业数据
  ↓
股票 / ETF / 基金分析快照
  ↓
策略信号 + 分级建议
  ↓
个人持仓复盘
  ↓
日报 / AI 解释
```

当前系统已经具备：

- 观察资产：股票、ETF、场外基金都能进入观察池。
- 个人持仓：支持默认个人组合，区分真实持仓和观察对象。
- 股票分析：围绕趋势、行业、估值、质量、资金和风险。
- ETF 分析方向：应使用价格、趋势、波动、回撤、流动性和主题暴露。
- 场外基金分析：围绕净值表现、收益、回撤、波动和风险等级。
- 分级建议：`继续观察`、`买入关注`、`持有`、`减仓关注`、`卖出关注`。
- 市场与行业全景：覆盖热门、冷门、轮动、防守、过热方向。
- 日报与 AI 解释：解释规则生成的建议，不直接替用户下单。

## 当前短板

当前主要问题不是页面数量不够，而是底层模型还需要收敛。

关键短板包括：

- 缺少轻量迁移体系，继续加表和字段容易堆进初始化脚本。
- 缺少 `instrument` 资产主数据表，资产名称、类型和市场信息分散在多张表里。
- ETF 口径需要进一步独立，不能继续复用股票基本面、估值和公司质量评分。
- 股票基本面和估值目前还缺少真实财报、估值历史和公司质量数据层支撑。
- `investment_advice` 需要记录规则版本、来源快照、上一建议等级和变化原因。
- 行业映射不能长期依赖 `watchlist.group_name`，需要正式的行业/主题/指数/地区/风格暴露映射。
- 组合还缺少历史快照，难以做组合曲线、风险变化和阶段复盘。
- 前端仍偏多页面数据展示，需要升级为“今日概览 + 重要变化提醒”，让用户快速知道是否有值得介入的变化。
- 安全层仍停留在 Cloudflare 代理和 CORS 配置，需要补访问前鉴权和部署边界。
- 宏观数据、新闻事件、政策变化和外部市场尚未形成正式上下文层。

## 业务终局

长期方向不是自动交易系统，也不是单纯行情工具。

目标是：

```text
本地优先的个人投资决策与复盘系统
```

它应该长期回答：

- 我正在研究什么资产？
- 我真实持有什么资产？
- 每个资产暴露在哪些行业、主题、指数、地区或风格？
- 今天哪些资产需要关注？
- 系统为什么给这个建议？
- 哪些建议、风险或重要变化需要我介入？
- 我的决策后来结果如何？
- 即使我没有每天处理建议，系统是否仍在自动跟踪后续表现？
- 我的组合风险是否改善？
- 股票财报和估值变化是否影响公司质量判断？
- 宏观、政策和新闻事件如何影响我的资产和持仓？

## 基础安全与访问保护

个人投资系统会包含持仓、建议、复盘、策略配置和未来的决策记录，因此需要轻量访问保护。

第一阶段不做复杂多用户登录，优先使用 Cloudflare Access 作为外层访问保护：

```text
浏览器
  ↓
Cloudflare Access 登录鉴权
  ↓
前端 / API
  ↓
Personal Invest 服务
```

需要区分：

- Cloudflare 普通代理 / CDN / WAF：解决域名代理、HTTPS、缓存和基础防护，不等于登录。
- Cloudflare Access：在访问应用前做身份校验，可以限制只有授权邮箱、GitHub 或 Google 账号访问。

第一阶段保护范围：

- `invest.chalme.indevs.in`
- `api.chalme.indevs.in`

部署边界：

- 前端和 API 都必须保护，不能只保护前端。
- 后端 `8000` 端口不能直接公网裸露，避免绕过 Cloudflare Access。
- 可以通过 Cloudflare Tunnel、防火墙、反向代理或安全组限制源站访问。
- `/health` 默认只保留最小健康信息，不暴露敏感配置。

后续增强：

- `P0-008`：Cloudflare Access 访问保护与部署边界。
- `P1-014`：应用内单用户登录。
- `P1-015`：API Token，用于脚本、自动化或移动端调用。
- 后续可增加审计日志，记录访问、关键配置修改和危险操作。

## 阶段一：模型收敛

目标：让资产、建议、暴露和组合历史变可信。

核心任务：

- 建立轻量数据库迁移体系。
- 引入 `instrument` 资产主数据。
- 修正 ETF 独立分析口径。
- 拆分 `advice_engine`，增加规则追溯。
- 新增资产行业/主题/指数/地区/风格映射。
- 新增组合历史快照。

阶段完成后，系统应能稳定回答：

- 每个资产是什么类型？
- 资产名称、市场、风险等级和来源在哪里统一维护？
- ETF 是否已经脱离股票基本面口径？
- 建议由哪条规则、哪版规则、哪份快照生成？
- 组合在股票、ETF、基金上的市值和风险如何变化？

## 阶段二：前端工作台升级

目标：把前端从多页面数据展示升级为“今日概览 + 重要变化提醒”。

前端升级不是单纯“更好看”，而是让用户一打开系统就能回答：

- 今天数据是否可信？
- 当前市场是否适合增加风险暴露？
- 是否出现需要立即关注的持仓风险？
- 哪些建议、风险或事件发生了重要变化？
- 为什么这件事值得关注？
- 明天最应该关注什么？
- 如果我不处理，系统如何继续自动跟踪后续结果？

核心原则：

- 风险优先：高风险、卖出关注、减仓关注、数据异常必须比普通机会更醒目。
- 低打扰优先：页面只在重要变化、风险升高或真实决策场景中提醒用户介入。
- 建议可操作：建议必须展示一句话结论、触发原因、关键证据、风险提醒和可选复核动作。
- 数据日期可见：市场、行业、资产、建议、持仓、日报都要展示数据日期或更新时间。
- 状态完整：加载、空状态、错误、任务运行中、任务失败都要有明确解释；无重要变化时应明确显示“暂无需要立即处理事项”。
- 移动端可浏览：移动端不一定完成复杂录入，但必须能浏览今日风险、建议和组合状态。

后续可拆分为：

- `P1-010`：前端信息架构重组，将导航从页面列表升级为工作流：今日、观察、持仓、建议、复盘、宏观、设置。
- `P1-011`：Dashboard 升级为今日概览与重要变化提醒，突出数据状态、风险变化、关键建议和明日关注。
- `P1-012`：持仓页升级为组合风险页，强化资产暴露、建议状态和复核动作。
- `P1-013`：重要事项池与低摩擦复盘页，集中展示重要变化、处理状态、自动跟踪结果和周/月复盘入口。
- `P2-002`：响应式和主题系统，补齐移动端、亮/暗主题和密度设置。

执行 UI 升级时应使用现有 frontend/design skill 做审计和预检，但不能套用营销页或 landing page 风格。本项目应保持产品 UI、金融仪表盘和表格工作流的专业密度。

## 阶段三：低摩擦决策复盘闭环

目标：从“系统给建议”升级为“系统自动沉淀建议、风险和结果，用户只在重要变化或真实交易时介入”。

建议新增核心对象：

- `review_task`：重要事项池，不是每日待办。系统只把重要建议变化、风险升高、数据异常、真实交易或需要用户确认的事项放入池中。
- `decision_record`：记录用户买入、持有、减仓、卖出或暂不处理的决策。
- `decision_outcome`：跟踪决策后 1 天、1 周、1 月的结果。
- `investment_policy`：记录个人风险偏好、仓位纪律和投资约束。

`review_task` 第一版建议状态：

| 状态 | 含义 |
|---|---|
| `OPEN` | 仍需要关注的重要事项 |
| `ACKNOWLEDGED` | 用户已看过，但不一定需要立即处理 |
| `SNOOZED` | 用户选择稍后再看 |
| `RESOLVED` | 用户已经处理，或事项已经通过数据变化解除 |
| `AUTO_EXPIRED` | 用户未处理，但系统判断建议、风险或事件已经失效 |

这一阶段系统应回答：

- 哪些重要事项需要用户介入，哪些只是自动沉淀？
- 用户未处理建议时，后续价格、净值、风险和建议等级如何变化？
- 用户为什么买入、持有、减仓或卖出？
- 用户为什么确认、延后或忽略某条重要事项？
- 建议之后资产表现如何？
- 用户决策是否改善了组合风险？
- 哪些行为模式值得改进？

复盘节奏应以周报和月报为主，而不是要求用户每天打卡。日报负责记录当天变化，周报和月报负责总结建议命中、风险变化、组合暴露和用户真实决策结果。

## 阶段四：股票基本面与财报分析层

目标：让股票建议不只依赖价格、行业和风险事件，而是能解释公司质量、财报变化和估值水平。

该阶段只适用于 `STOCK`。ETF、LOF 和场外基金不使用公司财报、公司估值或公司质量评分。

建议新增数据层：

- 财报数据：利润表、资产负债表、现金流量表、分红和回购。
- 财务指标：营收增速、净利润增速、毛利率、净利率、ROE、ROA、负债率、经营现金流和自由现金流。
- 估值数据：PE、PB、PS、股息率、历史分位、行业分位和估值区间。
- 财报事件：业绩预告、定期报告、利润大幅变化、现金流异常、资产减值、审计意见和管理层变化。

建议新增核心对象：

- `financial_statement_snapshot`
- `financial_indicator_snapshot`
- `valuation_snapshot`
- `financial_event`

阶段完成后，系统应能回答：

- 这家公司最近几个季度质量是变好还是变差？
- 利润增长是否有现金流支撑？
- 负债、应收、存货、商誉是否出现异常？
- 当前估值相对历史和行业是否偏高？
- 财报变化是否应触发风险事件或重要事项池？
- `fundamental_score`、`valuation_score` 和 `risk_score` 的依据是什么？

原则：

- 财报和估值先结构化，再进入股票分析评分。
- AI 负责解释财报变化和评分原因，不直接根据财报自由生成买卖建议。
- 财报异常可以进入 `risk_event` 和 `review_task`，但仍由规则决定建议等级。

## 阶段五：宏观新闻上下文

目标：让系统理解价格之外发生了什么。

建议新增数据层：

- 宏观指标：利率、通胀、PMI、汇率、社融、国债收益率、就业、消费、地产等。
- 新闻事件：公司新闻、行业新闻、政策事件、基金经理变动、重大风险事件。
- 外部市场：美股、港股、商品、黄金、原油、美元指数、海外利率等。
- 事件影响映射：把宏观或新闻事件映射到行业、主题、资产和持仓。

建议新增核心对象：

- `macro_indicator_snapshot`
- `news_event`
- `event_asset_impact`
- `ai_time_window_summary`

原则：

- 新闻和宏观数据先结构化。
- 规则先判断影响范围和风险等级。
- AI 再解释影响，不直接根据新闻自由生成买卖结论。

## 阶段六：个性化策略

目标：从通用规则升级为个人投资纪律系统。

个性化规则应覆盖：

- 单一资产最大仓位。
- 行业、主题、指数、地区和风格最大暴露。
- 股票、ETF、基金的偏好权重。
- 最大可接受回撤。
- 市场弱势时降低买入关注权重。
- 亏损扩大、盈利回撤、仓位过高时的复核要求。

这一阶段系统应回答：

- 这个资产本身是否值得关注？
- 它是否适合当前组合？
- 它是否符合用户的风险偏好？
- 当前是否应该提高、降低或维持关注优先级？
- 用户过去在类似场景下表现如何？

## 阶段七：长期产品形态

长期产品应形成以下核心页面和能力：

- Dashboard：今天是否有重要变化，哪些风险和建议值得关注。
- 观察池：长期研究对象、关注理由、当前建议和下一步观察点。
- 持仓页：组合暴露、盈亏、集中度、风险和操作复核。
- 建议页：分级建议、触发规则、关键证据、可选复核动作和自动跟踪状态。
- 复盘页：重要事项、用户决策、后续结果、周/月总结和行为模式。
- 财报页：公司质量、财报趋势、估值分位、财报事件和 AI 财报解释。
- 宏观新闻页：外部环境、事件影响和资产暴露。
- 日报：今日解释和明日关注。
- 月报 / 季报：阶段性市场、组合、建议和决策总结。
- AI：解释过去、总结阶段、生成明日关注、做情景推演。

## AI 边界

AI 的角色是解释器、总结器和复盘助手，不是黑盒预测器。

AI 应遵守：

- 主要解释过去。
- 重点解释今天。
- 未来 1 个交易日只给关注清单。
- 未来 1 周 / 1 月只做情景推演和风险展望。
- 不做确定性涨跌预测。
- 不绕过规则直接生成买卖建议。
- 输出建议时必须引用规则依据、关键指标、数据日期和风险说明。

推荐时间窗口：

| 时间窗口 | AI 角色 |
|---|---|
| 今日 | 解释市场、行业、资产、持仓和建议变化 |
| 明日 | 给出关注清单和复核动作 |
| 最近 1 周 | 总结短期趋势、风险累积和建议变化 |
| 最近 1 月 | 总结阶段性市场结构、组合表现、财报变化和决策效果 |
| 未来 1 周 / 1 月 | 做情景推演，不做确定性预测 |

## 不做什么

长期也应保持边界：

- 不做自动交易。
- 不替用户直接下单。
- 不承诺预测涨跌。
- 不把新闻摘要当成投资结论。
- 不把 AI 输出当成不可追溯建议。
- 不把 Cloudflare 普通代理当成登录保护。
- 第一阶段不做多账户、券商同步和实时行情流。

## 当前优先级

在长期规划下，当前优先级仍然是模型收敛：

1. 迁移体系。
2. `instrument` 资产主数据。
3. ETF 独立分析口径。
4. `advice_engine` 和规则追溯。
5. 资产行业/主题暴露映射。
6. 组合历史快照。

完成这些后，再进入前端工作台升级、低摩擦决策复盘闭环、股票基本面与财报分析、宏观新闻上下文和个性化策略。
